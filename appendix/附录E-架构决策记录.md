# 附录 E：架构决策记录

> 本附录记录 Claude Code 设计过程中的关键架构决策及其理由。

---

## E.1 技术选型

### E.1.1 选择 Bun 作为运行时

**日期**：2023 年中

**状态**：已采纳

**背景**：
Claude Code 需要快速的启动时间和高效的 JavaScript 执行。

**决策**：
使用 Bun 作为主要运行时和构建工具。

**理由**：
- 启动速度比 Node.js 快 3-4 倍
- 原生 TypeScript 支持，无需编译
- 内置 bundler、package manager
- 兼容 Node.js 模块

**后果**：
- CLI 启动时间从 ~500ms 降低到 ~100ms
- 构建流程简化，无需额外工具

**替代方案**：
- Node.js + ts-node：启动慢
- Deno：生态系统不如 Node.js 成熟

---

### E.1.2 自研 Ink 框架

**日期**：2023 年中

**状态**：已采纳

**背景**：
需要精确控制终端 UI，现有库无法满足需求。

**决策**：
自研 Ink 终端渲染框架。

**理由**：
- React 组件模型：需要 Fiber 架构支持
- 精确光标控制：需要逐像素控制
- 焦点管理：复杂的焦点状态
- Vim 模式：需要 vi 式输入处理

**后果**：
- 完全控制渲染流程
- 无外部框架依赖
- 需要维护自己的 React Reconciler

**替代方案**：
- blessed：API 不够 React 风格
- React Ink：功能不够丰富

---

### E.1.3 使用 Zod 进行验证

**日期**：2023 年中

**状态**：已采纳

**背景**：
需要同时进行编译时和运行时类型验证。

**决策**：
使用 Zod v4 作为主要验证库。

**理由**：
- TypeScript 类型自动推导
- 运行时验证 + 类型安全
- 链式 API，简洁易用
- 清晰的错误消息

**后果**：
- 所有工具输入都有 Schema 定义
- 类型安全贯穿整个系统

**替代方案**：
- Yup：TypeScript 支持不够
- Joi：不支持类型推导
- 自研验证：维护成本高

---

### E.1.4 采用 MCP 协议

**日期**：2024 年初

**状态**：已采纳

**背景**：
需要标准化外部工具和资源的集成方式。

**决策**：
全面采用 MCP (Model Context Protocol) 作为外部集成协议。

**理由**：
- 标准化：统一协议替代分散实现
- 可发现性：自动发现工具和资源
- 安全性：权限隔离
- 互操作：任何 MCP 服务器都可被调用

**后果**：
- MCP 服务器生态可被直接利用
- 用户可扩展 Claude Code 能力
- 统一的工具调用方式

**替代方案**：
- 传统 API 集成：每个集成独立开发
- OpenAI 插件协议：不支持本地工具

## E.2 架构决策

### E.2.1 分层架构

**日期**：2023 年中

**状态**：已采纳

**决策**：
采用 8 层分层架构。

**层结构**：
```
CLI 入口 → REPL 界面 → 查询引擎 → 工具编排
→ API 通信 → MCP 协议 → 状态管理 → 扩展机制
```

**理由**：
- 职责分离：每层只关注自己职责
- 可测试性：各层可独立测试
- 可替换性：可替换某层实现
- 可扩展性：易于添加新层

**后果**：
- 代码结构清晰
- 模块边界明确
- 便于团队分工

---

### E.2.2 双层状态管理

**日期**：2023 年中

**状态**：已采纳

**决策**：
使用 Bootstrap State (单例) + AppStateStore (响应式)。

**理由**：
- Bootstrap State：全局配置，一次初始化
- AppStateStore：频繁变化的 UI 状态
- 分离关注点：不变状态 vs 变化状态

**后果**：
- 状态管理清晰
- 性能优化方便
- 内存占用合理

---

### E.2.3 异步生成器模式

**日期**：2023 年中

**状态**：已采纳

**决策**：
QueryEngine 使用 AsyncGenerator 实现流式响应。

**理由**：
- 边生成边输出：用户体验更好
- 可中止：用户可中断操作
- 内存效率：无需等待全部完成

**后果**：
- 实时响应式输出
- 支持长时间操作的中止
- 流式 API 调用

---

### E.2.4 分区并发执行

**日期**：2023 年中

**状态**：已采纳

**决策**：
工具执行采用分区并发策略。

**理由**：
- 独立工具可并行：提高性能
- 有依赖工具顺序执行：保证正确性
- 分区策略：最优并行度

**后果**：
- 工具调用性能提升
- 无依赖工具不互相等待

## E.3 安全决策

### E.3.1 权限系统设计

**日期**：2023 年中

**状态**：已采纳

**决策**：
实现基于规则引擎的权限系统。

**规则结构**：
```typescript
interface PermissionRule {
  action: string
  resource?: Pattern
  effect: 'allow' | 'deny' | 'confirm'
  conditions?: Condition[]
}
```

**理由**：
- 灵活性：可配置规则适应不同场景
- 安全性：默认拒绝，精确允许
- 可扩展：易于添加新规则类型

**后果**：
- 7 种权限模式适应不同用户
- 敏感操作必须确认
- 完整审计日志

---

### E.3.2 保护路径机制

**日期**：2023 年中

**状态**：已采纳

**决策**：
实现系统级保护路径。

**保护路径**：
- `/etc/*` - 系统配置
- `~/.ssh/*` - SSH 密钥
- `~/.aws/*` - 云凭证
- `**/.env` - 环境变量
- `**/secrets.*` - 密钥文件

**理由**：
- 防止误操作导致系统损坏
- 保护敏感凭证
- 符合最小权限原则

**后果**：
- 敏感文件访问被阻止
- 用户需明确授权才能访问

## E.4 性能决策

### E.4.1 上下文压缩策略

**日期**：2023 年中

**状态**：已采纳

**决策**：
实现多级上下文压缩。

**压缩级别**：
- 警告 (70%)：提示用户
- 压缩 (85%)：自动摘要历史
- 紧急 (95%)：强制压缩

**理由**：
- 200K token 限制需要管理
- 用户不应手动管理上下文
- 保留关键信息

**后果**：
- 无需手动管理上下文大小
- 对话可一直持续
- 关键信息不会被丢弃

---

### E.4.2 工具结果缓存

**日期**：2024 年初

**状态**：已采纳

**决策**：
实现工具结果缓存。

**缓存策略**：
- 文件内容：TTL 5 分钟
- 目录列表：TTL 1 分钟
- Glob 结果：TTL 30 秒

**理由**：
- 重复读取代价高
- 文件系统调用延迟
- 上下文不重复包含相同内容

**后果**：
- 重复工具调用更快
- 减少 API token 使用
- 保持上下文高效
